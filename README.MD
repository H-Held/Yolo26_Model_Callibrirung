> **🧠 Kern-Philosophie: „Lieber zuviel als zuwenig“**  
> Das Modell soll **mehr Objekte erkennen** als nötig — False Positives sind
> akzeptabel, False Negatives nicht. Der optimale Confidence-Schwellwert wird
> so gewählt, dass der **Recall maximiert** wird, solange die Precision
> **nicht unter 50 %** fällt. Lieber 10 Fehlalarme als 1 übersehenes Objekt.

Automatische Evaluierung von YOLO-Modellen gegen COCO-annotierte Testdaten.  
Berechnet alle Metriken und schreibt eine sauber formatierte `.env`-Datei.

## 📋 Erforderliche Datenformate

### 1. Modell – YOLO `.pt`

Jedes YOLOv5/v8/v11/... oder YOLO26 `.pt`-File.  
Das Script erkennt automatisch das erste `.pt` in `model/`.  
Override mit `--model`:

```bash
python programmes/model_evaluator.py --model model/yolo26_face_lp.pt
```

### 2. Annotationen – COCO 1.0 JSON

```json
{
  "images": [
    { "id": 0, "file_name": "bild1.jpg", "width": 3072, "height": 4096 }
  ],
  "annotations": [
    {
      "id": 0,
      "image_id": 0,
      "category_id": 2,
      "bbox": [1904, 737, 423, 518],
      "area": 219095,
      "iscrowd": 0
    }
  ],
  "categories": [
    { "id": 2, "name": "face" },
    { "id": 4, "name": "license-plate" }
  ]
}
```

> ⚡ **Wichtig:** COCO-Kategorien werden **automatisch per Name** mit den Modell-Klassen
> gematcht (normalisiert: lowercase, `-` → `_`). Kategorien, die das Modell nicht kennt
> (z. B. `car`, `person`), werden ignoriert und beim Lauf ausgegeben.

### 3. Bilder

Unterstützt: `.jpg`, `.jpeg`, `.png`  
Müssen im `test_data/images/`-Ordner liegen und mit `file_name` in der COCO-JSON übereinstimmen.

## 🧪 Verwendung

### Standard (liest Pipeline aus `.env`)

```bash
python programmes/model_evaluator.py
```

Liest `TILES_USED` und `COMPRESSED` aus der `.env` und führt **nur diese eine**
Pipeline aus. Schreibt Ergebnisse zurück in die `.env`.

### Auto-Modus (alle 4 Pipelines testen)

```bash
python programmes/model_evaluator.py --pipeline auto
```

Probiert alle 4 Modi, zeigt Vergleichstabelle, nimmt den besten (nach mAP@50).

### Bestimmte Pipeline erzwingen

```bash
python programmes/model_evaluator.py --pipeline tiles_compress
python programmes/model_evaluator.py --pipeline tiles
python programmes/model_evaluator.py --pipeline compress
python programmes/model_evaluator.py --pipeline direct
```

### Anderes Modell / CPU

```bash
python programmes/model_evaluator.py --model model/v2.pt
python programmes/model_evaluator.py --device cpu
```

### Tile/Compress-Parameter anpassen

```bash
python programmes/model_evaluator.py --tile-size 1024 --tile-overlap 10 --compress-size 512
```

## 🔧 Die 4 Pipeline-Modi

| Modus | TILES_USED | COMPRESSED | Beschreibung |
|---|---|---|---|
| `tiles_compress` | true | true | Bild → 1280px Tiles → je auf 640px komprimieren → YOLO → NMS |
| `tiles` | true | false | Bild → 1280px Tiles → YOLO → NMS |
| `compress` | false | true | Bild → auf 640px verkleinern → YOLO |
| `direct` | false | false | Bild in Originalgröße → YOLO |

## 📊 Ausgabe (`.env`)

Nach jedem Lauf wird die `.env` sauber formatiert überschrieben:

```dotenv
# =============================================================================
# Release metadata — auto-generated by model_evaluator.py
# =============================================================================

# --- Identity / version ---
VERSION=v1.1.0
lowest_compatible_SoftwareVersion=v1.1.0
BASE_MODEL=yolo26
MODEL_SIZE=l

# --- Tiling ---
TILES_USED=true
TILE_SIZE=1280
TILE_OVERLAP_PCT=20.0

# --- Compression ---
COMPRESSED=true
COMPRESSION_FORMAT=downscale
COMPRESSION_QUALITY=640

# --- Classes ---
CLASSES=0:face,1:license-plate
CONF_THRESHOLDS=0:0.709,1:0.159
NUM_CLASSES=2

# --- Evaluation metrics ---
METRIC_FACE_RECALL=0.005
METRIC_FACE_PRECISION=1.0
METRIC_LICENSE_PLATE_RECALL=0.375
METRIC_LICENSE_PLATE_PRECISION=0.545
METRIC_MAP50=0.203
METRIC_MAP50_95=0.108
```

## 🧠 Threshold-Logik: Recall-First („Lieber zuviel als zuwenig“)

Das ist das **zentrale Prinzip** des Evaluators. Für jede Klasse wird der
Confidence-Schwellwert so gewählt:

```
┌─────────────────────────────────────────────────────────┐
│  Für jeden Schwellwert 0.01 … 0.95 (Schritt 0.01):     │
│    → Precision & Recall berechnen                       │
│                                                         │
│  🏆 Optimaler Schwellwert =                             │
│     Höchster Recall, bei dem Precision noch ≥ 0.5 ist   │
│                                                         │
│  Falls keine Stufe Precision ≥ 0.5 erreicht:            │
│     → Schwellwert mit der höchsten Precision nehmen     │
└─────────────────────────────────────────────────────────┘
```

**Konkretes Beispiel (license-plate):**

| Confidence | Recall | Precision | Bewertung |
|---|---:|---:|---|
| 0.01 | 0.50 | 0.30 | ❌ Precision zu niedrig |
| 0.10 | 0.43 | 0.48 | ❌ Precision knapp unter 0.5 |
| **0.16** | **0.375** | **0.545** | ✅ **Optimal! Höchster Recall mit P≥0.5** |
| 0.66 | 0.25 | 0.571 | Precision gut, aber weniger erkannt |

→ Ergebnis: `CONF_THRESHOLDS=…,1:0.159` — **mehr Detections, weniger übersehene Objekte.**

Die Precision-Grenze von 0.5 stellt sicher, dass nicht völlig wahllos erkannt
wird — aber der Fokus liegt klar auf **maximaler Erkennungsrate**.

## 📈 Metriken

| Metrik | Beschreibung |
|---|---|
| `METRIC_<NAME>_RECALL` | Recall pro Klasse beim optimalen Schwellwert |
| `METRIC_<NAME>_PRECISION` | Precision pro Klasse beim optimalen Schwellwert |
| `METRIC_MAP50` | mAP@50 (11-Point Interpolation) |
| `METRIC_MAP50_95` | mAP@50-95 (Approximation ×0.53) |
